JS的作用域以及命名空间

发表于 2015-11-12   |   分类于 JavaScript   |     |  阅读量 次

之前写过一篇博客，总结了一下JS变量。前段时间开始试着写自己的库的时候，有遇到了一些问题，发现自己有些东西还没有完全的刻在脑子里，这里再总结一下。

JS的作用域

JS是没有块级作用域的（这里不说ES6）。JS中，函数，变量的作用域以及作用域链是在代码解析阶段就完成的，不是在执行阶段。用以下的例子就可以说明: 

function f(){
    var bb=”123”;
    var aa=function(){
        console.log(bb);
    }
    return aa;
}
function cc(para){
    var bb=para;
    var func=f();
    func();
}
cc(“111”);

这段代码的输出是:”123”，而不是”111”。这是因为在声明函数的时候，函数f及其内部aa指向的匿名函数的作用域、作用域链就已经确定了。该匿名函数的作用域链为：aa->f->window.
在cc函数中，func函数指向的也是aa指向的函数，是同一个函数。所以在cc执行的时候，func指向的那个匿名函数是读不到func外部，cc内部的那个bb变量的，只能沿着其作用域链aa->f->window向上走，在f函数的作用域内找到了bb=”123”.将其输出.

JS的命名空间

说到自己写的那个很搓的库myComponent.js，开始写的时候，我是按照impress.js那个模式来写的。库代码的模板为: 

(function (window,document){
    …
    …
    var myComponent = window.myComponent = (function () {
        var init = function () {
            …
        }
        return {
            init: init
        }
    })()
})(window,document)

而使用这个库的时候，只需要在html文件中，引入myComponent.js并且使用myComponent.init()就行了。像下面这样:

其实，这里就涉及到了命名空间。
使用一个立即执行的匿名函数来完成整个库的逻辑，在这个匿名函数外部是访问不到函数内部的变量的，然后在window对象上「注册」一个变量myComponent,而这个myComponent其实是一个接口，使得可以在html文件中调用这个库。这样封装的好处就是:库文件内的代码和外部的代码完全隔离，互不影响，不会出现因变量名相同而导致的代码混乱。
