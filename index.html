移动web开发学习笔记-2

发表于 2016-04-08 | 分类于 移动web |

通过上一篇文章，已经知道了css中的1px并不等于设备的1px。在移动设备上进行网页的重构或开发，首先得搞明白的就是移动设备上的viewport了，只有明白了viewport的概念以及弄清楚了跟viewport有关的meta标签的使用，才能更好地让我们的网页适配或响应各种不同分辨率的移动设备。

什么是viewport

这里，先将viewport理解为视口的意思吧。对于用户来说，希望在移动设备上的浏览器上也可以正常的浏览网站，即使是那些不是在移动设备上设计的网站。那么问题来了，就人的直观上来说，跟PC相比，手机的屏幕这么小（viewport小），怎么样让那些为PC设计的网站完美的在手机上显示呢？

这就需要将页面等比例的缩小。缩小页面的过程中，会有一个问题，如果直接将网页按照移动设备的屏幕大小来进行缩放，那么很多网页就会因为viewport太窄而出现布局错乱，字太小，看不清等。

移动设备浏览器就为了我们做了两件事：

  • 先将页面渲染在一个较大的viewport中，保证页面排版正确。
  • 然后再进行缩放，让用户能够看得正确。

我们将用来排版的那个大的viewport叫做layout viewport。这个layout viewport的宽度可以使用document.body.clientWidth来获取。这个viewport是一个虚拟的概念。一般情况下，layout viewport都是大于移动设备的屏幕宽度的。每种移动设备的layout viewport都不一样。iphone是980px，安卓设备各有差异。

我们将移动设备浏览器可视区域的宽度叫做visual layout。这个visual layout的宽度可以通过window.innerWidth来获取。需要注意的是，只有在正确设置了meta标签后，这里的innerWidth的值才跟设备的可视区域相等。在没有设置的meta标签时，这个值等于网页文档的width值，由网页内容决定。

为了让页面完美显示（不需要用户缩放和横向滚动条就能正常的查看网站的所有内容；显示的文字的大小是合适的），我们需要对上述的viewport进行改造。而改造viewport，就需要设置meta标签。

利用meta标签对viewport进行控制

在开发移动设备的网站时，最常见的的一个动作就是把下面这个东西复制到我们的head标签中：

<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=0">

该meta标签的含义就是：visual viewport = 设备可视区域宽度 = layout viewport

这是最适合移动设备的viewport的设置。

下面是meta viewport的详细规范。

width           设置layout viewport的宽度。值为一个正整数或者『device-width』。
                device-width表示与设备宽度同宽。
initial-scale   设置页面的初始缩放值。
minimum-scale   设置页面的最小缩放值。
maximum-scale   设置页面的最大缩放值。
user-scalable   是否允许用户进行缩放，值为『yes』或者『no』,1或者0

下面有几个地方需要注意一下：

  • 在没有设置缩放，只设置了width属性时，如果页面的宽度大于了设备的宽度，则window.innerWidth的值为页面的宽度，也就是可视区域的宽度，此时的缩放比为：window.innerWidth/document.body.clientWidth，浏览器会自动进行缩放，使得不会出现横向滚动条。
  • 在同时设置了缩放比，以及width属性的情况下，如果页面的宽度大于了设备的宽度，并且此时按照widow.innerWidth/document.body.clientWidth计算出来的缩放比与设置的缩放比不一致时，会按照设置的initial-scale进行缩放，会出现横向滚动条。

其实绕了这么多，我也晕了。最后总结起来就是：

width = device-width  <==>  layout port = 设备宽度
initial-scale = 1     <==>  visual port = layout port

最常用的，最佳的viewport的设置为：

<meta name="viewport" content="width=device-width, initial-scale=1.0, user-scalable=0">

移动web开发学习笔记-1

发表于 2016-03-24 | 分类于 移动web |

所谓移动web开发，就是开发适配于移动端、主要是手机端的web页面，也就是常说的H5页面。移动web开发跟pc端的web开发有很大的不同，坑更多。先从基础入手吧。

像素

像素分为了逻辑像素和抽象像素。

px 逻辑像素，浏览器使用的抽象单位。

dp 设备无关像素，即物理像素。注意，dip 也表示同样的概念

两个的关系是由一个叫做 dpr 的值来决定的。dpr 被成为设备像素缩放比。


计算公式

1px= 1dp / (dpr)²

手机屏幕

手机屏幕尺寸：屏幕的对角线的长度，单位是英寸。

屏幕分辨率：指在横纵向上的像素点数，单位是px，1px = 1个像素点。一般以纵向像素×横向像素来表示一个手机的分辨率，如1960×1080。（这里的1像素指的是物理设备的一像素点）。

屏幕像素密度：屏幕上每英寸可以显示的像素点的数量，单位是 ppi (pixels per inch)。注意：dpi 也表示一样的概念。一般android设备使用该单位


计算像素密度的公式为（以1960×1080像素5英寸为例）

( √（1920²+1080²）) / 5 = 440 ppi // 先求出对角线像素点数，再除以手机频幕尺寸

dpr(设备缩放比)的确定

android和iphone都是根据ppi来确定dpr的，但是两者的标准不一样。


android的标准如下：

名称                dpi范围                dpr的值
ldpi                <120                     1
mdpi                120 - 160                1
hdpi                160 - 240                1.5
xhdpi               240 - 320                2
xxhdpi              320 - 480                3
xxxhdpi             480 - 640                4

iphone的标准如下：

名称                         dpr的值
非高清屏(non-retina)             1
高清屏（retina)                  2
超高清屏 (retina HD)             3

总结

我们在进行移动web开发的时候，需要先进行逻辑像素值的确定，即需要知道在浏览器环境下的px与设备的px是什么关系。下面以iphone5为例。

  • iphone5的设备分辨率为 1136dp × 640dp.

  • 计算其ppi.( √（1136²+640²）) / 4 = 326 ppi.

  • 在ios的标准中，iphone5的屏幕为retina，其dpr值为2.（其实对于iphone，不用计算其ppi，直接根据屏幕类型就能确定dpr）.

  • 根据公式 1px= 1dp / (dpr)² , iphone5的屏幕的逻辑像素为：320px × 568px

那么问题来了，当我在页面上声明一个 320px × 568px 大小的div时，并不能占满整个屏幕，这是怎么回事呢？其实是因为没有正确设置viewpoint.

js中的私有变量

发表于 2015-12-22 | 分类于 JavaScript |

在面向对象编程的过程中，难免会遇到一些场景，对象需要保存其私有的变量，不被外界所修改。这里总结一下JavaScript中，关于对象私有变量的问题。

Javascript中的对象

在JS中生成一个对象，构造函数是一个较为常用的方法。

function Person(name,age){ 
    this.name=name;
    this.age=age;
}

var coy=new Person("coy",24);
console.log(coy);//{name:'coy',age:24}

在上面的例子中，构造了一个Person对象，coy是Person的一个实例。JS中构造对象的方法很多，这里就不再赘述了。

JS对象中的公共变量

function Person(name,age){ 
    this.name=name;
    this.age=age;
}

var coy=new Person("coy",24);
console.log(coy.name);//coy
console.log(coy.age);//24;
coy.name="ryan";
console.log(coy.name)//ryan

继续看这个例子，name和age作为实例coy的属性，是可读可写的。

接着上面的代码，再看下面的代码：

Person.type={type:"animal"}；
console.log(coy.type);//undefined
console.log(Person.type);//{type:"animal"}

var alec=new Person("alec",22);
console.log(alec.constructor.type);//{type:"animal"}
console.log(coy.constructor.type);//{type:"animal"}
console.log(alec.constructor.type===Person.type);//true
console.log(alec.constructor.type===coy.constructor.type);//true

直接在Person对象上赋值，Person对象的实例无法访问到type属性。而必须通过Person直接访问,而且每一次访问都是访问的同一个type,这里的type属性可以称为公共静态变量。当然，对于类似这样的函数，也被称为公共静态方法。

虽说这样的公共静态变量非常方便，但是他可以直接读写，安全性是个问题。那么如何将其“保护”起来呢？

JS对象中的私有变量

为对象构造私有变量，有以下的方法。

构造函数法

在构造函数中，使用var来声明的变量，就是对象的私有变量。看下面的代码：

function Person(name,age){ 
    this.name=name;
    this.age=age;
    var sex="boy";
}
var coy=new Person("coy",24);
var alec=new Person("alec",22); 

console.log(Person.sex);//undefined
console.log(coy.sex);//undefined

可见，sex属性是无法被直接访问的。怎么样访问呢？

function Person(name,age){ 
    this.name=name;
    this.age=age;
    var sex="boy";

    this.getSex=function(){
        return sex;
    }
    this.setSex=function(a){
        sex=a;
    }
}
var coy=new Person("coy",24);
var alec=new Person("alec",22);
alec.setSex("girl");
console.log(coy.getSex());//boy
console.log(alec.getSex());//girl

这里，在对象的构造函数内，声明了一个getSex的方法和setSex方法。通过这两个方法就可以访问到sex这个属性了。这里是由于JS的闭包和作用域机制。具体就不展开了。这里需要注意的是，Person原型上的方法是无法访问到sex这个属性的。像这种在构造函数内部定义的，可以访问到私有变量的方法被称为特权方法。

这里有一个问题就是：每次实例化一个对象，就会声明一遍这些静态变量和特权方法。这是一种对资源的浪费。

立即执行函数法

先看下面的代码：

(function(){

    var type="animal";

    Person=function(name,age){
        this.name=name;
        this.age=age;
    }

    Person.prototype.setType=function(a){
        type=a;
    }
    Person.prototype.getType=function(){
        return type;
    }
})()

var coy=new Person("coy",24);
var alec=new Person("alec",22);
console.log(coy.getType());//animal
console.log(alec.getType());//animal

alec.setType("girl");
console.log(coy.getType());//girl
console.log(alec.getType());//girl
  • 使用一个立即执行的匿名函数将对象的声明代码包裹起来。这样，形成了一个特定的作用域。
  • 在该作用域中，使用var来声明了私有变量type。
  • 注意，对象的构造函数Person是直接声明的，没有使用var来声明，使得该构造函数成为了一个全局变量，可以在该特定作用域外访问。
  • 特权方法在构造函数的原型链上声明，对象实例化时，不用每次都声明一下这些特权方法了。
  • 这种创建的私有变量是所有实例共享的。某一个实例对私有变量进行了修改，会影响所有的实例。这是因为JS的作用域链和闭包的缘故。

JS中还有其他很多的设计模式都会涉及到私有变量的问题。Keep learing…

JS的作用域以及命名空间

发表于 2015-11-12 | 分类于 JavaScript |

之前写过一篇博客，总结了一下JS变量。前段时间开始试着写自己的库的时候，有遇到了一些问题，发现自己有些东西还没有完全的刻在脑子里，这里再总结一下。

JS的作用域

JS是没有块级作用域的（这里不说ES6）。JS中，函数，变量的作用域以及作用域链是在代码解析阶段就完成的，不是在执行阶段。用以下的例子就可以说明: 

function f(){
    var bb=”123”;
    var aa=function(){
        console.log(bb);
    }
    return aa;
}
function cc(para){
    var bb=para;
    var func=f();
    func();
}
cc(“111”);

这段代码的输出是:”123”，而不是”111”。这是因为在声明函数的时候，函数f及其内部aa指向的匿名函数的作用域、作用域链就已经确定了。该匿名函数的作用域链为：aa->f->window.
在cc函数中，func函数指向的也是aa指向的函数，是同一个函数。所以在cc执行的时候，func指向的那个匿名函数是读不到func外部，cc内部的那个bb变量的，只能沿着其作用域链aa->f->window向上走，在f函数的作用域内找到了bb=”123”.将其输出.

JS的命名空间

说到自己写的那个很搓的库myComponent.js，开始写的时候，我是按照impress.js那个模式来写的。库代码的模板为:

    …
    …

        var init = function () {
            …
        }
        return {
            init: init
        }
    })()
})(window,document)

其实，这里就涉及到了命名空间。
使用一个立即执行的匿名函数来完成整个库的逻辑，在这个匿名函数外部是访问不到函数内部的变量的，然后在window对象上「注册」一个变量myComponent,而这个myComponent其实是一个接口，使得可以在html文件中调用这个库。这样封装的好处就是:库文件内的代码和外部的代码完全隔离，互不影响，不会出现因变量名相同而导致的代码混乱。
